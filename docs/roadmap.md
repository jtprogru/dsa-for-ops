# Roadmap: куда развивать курс

Текущие лабораторные (`lab01`–`lab05`) дают крепкий академический фундамент: массив и стек, связный список, очередь и дерево, сортировки, поиск, хеш-таблица. Этот документ фиксирует, **куда курс растёт дальше** и **под кого**.

## Для кого этот курс

Целевая аудитория — будущие **SRE / DevOps / Ops**, которые хотят углубиться в программирование, но не понимают, с чего начать. Для них структуры данных и алгоритмы — не самоцель, а способ понять, **почему системы, которые они эксплуатируют, устроены именно так**: почему кэш вытесняет данные, почему шардирование ломается при добавлении ноды, почему сервисы стартуют в определённом порядке, почему запрос «разваливается» на масштабе.

Поэтому ведущий принцип развития курса: **каждая структура данных подаётся через инструмент, который инженер реально трогает руками.** Не «бинарная куча ради бинарной кучи», а «бинарная куча — это то, как планировщик выбирает следующую задачу».

## Принципы (что сохраняем)

Новые лабораторные следуют сложившемуся стилю репозитория:

- **«С нуля», без магии.** Минимум встроенных функций и сторонних библиотек — структуры реализуются руками, чтобы было видно, как они устроены внутри (как `lab05` обходится без `dict`).
- **Docstring с заданиями** в начале файла, осмысленные русскоязычные комментарии по ходу кода.
- **Меню / демонстрация** в `main()`, чтобы лабу можно было запустить и «пощупать» (`make run labNN`).
- **Тесты** в `tests/test_labNN.py` и **конспект** в `docs/labNN.md` с разбором сложности.
- **Общий код** выносится в `labs/common`.

Новое, что добавляем к стилю:

- Блок **«Где это в проде»** в каждом конспекте — 2–3 строки с конкретным инструментом (etcd, Redis, PostgreSQL, k8s, nginx), где структура работает «по-настоящему».

## Концептуальные пробелы текущего курса

| Пробел | Почему важен для Ops |
|---|---|
| **Графы** — нет вообще | зависимости сервисов, порядок запуска, обнаружение циклов, сетевая маршрутизация |
| **Куча / приоритетная очередь** — нет | планировщики (k8s, cron), top-K тяжёлых запросов, таймеры |
| **Big-O измеряется только в теории** | у Ops интуиция «почему это разваливается на масштабе» важнее формального доказательства |
| **Нет обработки реальных данных** | все лабы работают со случайными `int`; разбор логов и агрегация — это 80% повседневного скриптинга |
| **Рекурсия не выделена отдельно** | спрятана в обходе дерева; новичку нужен явный разбор (стек вызовов, база/шаг, переполнение) |

## Консолидация с `mti-oaip-lectures`

Параллельно существует репозиторий [`mti-oaip-lectures`](https://github.com/jtprogru/mti-oaip-lectures) — большой курс «Основы алгоритмизации и программирования»: 14 тем, параллельно **Python 3.14 и Go 1.23**, лекции + практика + экзаменационные билеты. По объёму он крупнее `mti-dsa` и содержит много материала, прямо релевантного аудитории SRE/DevOps (Go-трек, конкурентность, профилирование, логирование, тестирование).

**Решение:** оставляем единственный репозиторий — `mti-dsa`. Из `mti-oaip-lectures` переносим только релевантное (черри-пик), остальное архивируется вместе с исходным репозиторием.

!!! warning "Следствие — расширение области проекта"
    После переноса `mti-dsa` перестаёт быть только про «структуры данных и алгоритмы»: добавляется Python-фундамент для эксплуатации и **трек Go**. Это потребует обновить название/описание проекта, заголовок сайта (`mkdocs.yml`), README и, возможно, само имя репозитория (`dsa` уже не покрывает содержание). Открытый вопрос — см. конец документа.

### Что НЕ переносим (осознанно)

Чтобы репозиторий не превратился в «свалку всего курса», за бортом остаётся наименее релевантное для DSA + ops:

- **GUI** (тема 08: Tkinter, PyQt, web-UI) — не про эксплуатацию.
- **Django** (тема 11, лекции 2–3) — веб-фреймворк вне фокуса; при желании можно оставить ссылкой.
- **Экзаменационные билеты** и большой сквозной практический проект — привязаны к учебному курсу, а не к самообучению инженера.
- **Темы 01–02** (общая алгоритмизация, классификация языков) — частично дублируются конспектом `notes.md`; берём только то, чего нет.

При архивации `mti-oaip-lectures` стоит оставить в его README ссылку на `mti-dsa`, чтобы внешние ссылки на курс не упирались в тупик.

## План новых лабораторных

Порядок — по приоритету для целевой аудитории. Нумерация условная и может измениться. Часть материала для них уже готова в `mti-oaip-lectures` и переносится (см. карту выше), а не пишется с нуля.

### lab00 — Python для эксплуатации (фундамент «с чего начать»)

!!! note "Закрывает разрыв до структур данных"
    DSA — это середина пути. Человек из Ops споткнётся раньше: на файлах, аргументах CLI и исключениях. Без этой ступеньки «массив без `len`» выглядит абстрактным упражнением, а не подступом к рабочему скрипту.

- Работа с файлами (чтение/запись, контекстные менеджеры).
- Аргументы командной строки (`argparse`), коды возврата процесса.
- Исключения и обработка ошибок как часть контракта функции.
- Генераторы и итераторы (зацепка уже есть — самописный `CustomRange` в `labs/common/ranges.py`).
- **Где это в проде:** любой эксплуатационный скрипт — парсер конфига, обёртка над `subprocess`, healthcheck.

### lab06 — графы и топологическая сортировка (главный пробел)

- Граф через список смежности.
- Обходы: в ширину (BFS) и в глубину (DFS).
- Обнаружение цикла.
- Топологическая сортировка (порядок с учётом зависимостей).
- **Где это в проде:** порядок запуска зависимых сервисов (`systemd`, `docker-compose depends_on`), граф ресурсов Terraform, DAG в Airflow, обнаружение циклических зависимостей.

### lab07 — куча и top-K

- Бинарная min/max-куча на массиве.
- Приоритетная очередь поверх кучи.
- Задача top-K (K самых больших/тяжёлых элементов без полной сортировки).
- **Где это в проде:** планировщик задач (k8s scheduler), top-N самых медленных запросов в логе, таймерные колёса.

### lab08 — LRU-кэш

- Хеш-таблица (переиспользуем идеи `lab05`) + двусвязный список.
- Политика вытеснения Least Recently Used.
- **Где это в проде:** вытеснение в Redis/Memcached, кэш CDN, кэш страниц БД. Объясняет, почему кэш «промахивается» и зачем нужен правильный размер.

### lab09 — rate limiting

- Кольцевой буфер (ring buffer).
- Алгоритмы: token bucket, leaky bucket, sliding window.
- **Где это в проде:** троттлинг API, защита сервиса от перегрузки, лимиты на стороне клиента и сервера.

### lab10 — разбор логов (практический максимум)

- Потоковая обработка файла построчно (генераторы, чтобы не держать всё в памяти).
- Агрегация по полям (счётчики по статусам/IP/эндпоинтам).
- Перцентили латентности (p50/p95/p99).
- **Где это в проде:** ad-hoc анализ access-логов, подсчёт ошибок по сервисам, быстрая прикидка SLI без тяжёлых систем.

### lab11 — consistent hashing и bloom filter

- Consistent hashing: хеш-кольцо и виртуальные ноды.
- Bloom filter: вероятностная проверка принадлежности.
- **Где это в проде:** шардирование БД и балансировка (почему `hash % N` ломается при добавлении ноды), фильтр «точно нет» перед обращением к диску/сети (Cassandra, CDN).

## Порядок реализации

Сначала **перенос готового** из `mti-oaip-lectures` (дешевле, чем писать заново), затем **новые лабы**, затем **углубления**.

1. Зафиксировать roadmap (этот документ). ✅
2. **Перенос Python-фундамента** → собрать `lab00` (рекурсия, файлы, CLI, исключения) из лекций тем 04 и 07.
3. **Перенос Go-трека** → `src/golang/` + разделы `docs/go/basics` и `docs/go/advanced` из тем 13–14. Включить Go-примеры в CI (у lectures уже есть `go test ./...`).
4. **Перенос прод-тем** → `docs/ops/` (логирование, конкурентность, тестирование) и `docs/practice/` (git, шпаргалка).
5. Обновить «обвязку»: `mkdocs.yml` (nav + заголовок сайта), `README.md`, `pyproject.toml` (описание), при необходимости — имя репозитория.
6. `lab06 — графы` — закрывает самый большой DSA-пробел.
7. `lab10 — разбор логов` — самая прикладная лаба, переиспользует перенесённые regex/файлы.
8. Углубления существующих лаб (блоки «Где это в проде», бенчмарк в lab03, мостик lab05 → consistent hashing).
9. Остальные лабораторные (`lab07`–`lab09`, `lab11`) по мере интереса.
10. Архивировать `mti-oaip-lectures` со ссылкой на `mti-dsa` в README.

## Открытые вопросы

1. **Имя репозитория.** После переноса Go-трека и Python-фундамента `dsa` (data structures & algorithms) перестаёт описывать содержание. Кандидаты: переименовать (например, в курс «программирование для эксплуатации») или оставить `mti-dsa` как историческое имя и пояснить охват в описании.
2. **Глубина Go-трека.** Переносить Go как полноценный равноправный трек или как «приложение» к Python-курсу? Для SRE/DevOps аргумент в пользу полноценного трека сильный.
3. **Что делать с наименее релевантным** (Django, GUI, билеты): полностью отбросить или сохранить отдельной веткой/тегом в архиве `mti-oaip-lectures` на случай, если пригодится.
4. **Двуязычность лаб.** Делать ли для новых DSA-лаб параллельные реализации на Go (как в lectures), или DSA оставить только на Python, а Go держать отдельным треком.
